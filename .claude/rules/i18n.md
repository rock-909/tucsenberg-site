# i18n Rules

Use this file when changing translation keys, message loading, locale routing,
or next-intl usage.

For page prose and FAQ ownership, use `content.md`.

## Runtime truth

Runtime canonical source is physical message packs:

```text
messages/base/{locale}/messages.json
messages/profiles/b2b-lead/{locale}/messages.json
messages/profiles/catalog/{locale}/messages.json
```

Generated compatibility files are retired. Authoring and runtime both use the
physical packs above. Example primary sources:

```text
messages/base/en/messages.json
messages/profiles/b2b-lead/en/messages.json
messages/profiles/catalog/en/messages.json
```

Runtime loads composed pack imports through `src/lib/i18n/message-pack-loader.ts`.
Runtime loads bundled messages through `src/lib/i18n/load-messages.ts`.
Static and Storybook composition uses `src/lib/i18n/composed-messages.ts`.
Do not treat `public/messages/**` as an authoring or proof surface.

The composition graph is fixed: `base -> b2b-lead -> catalog`. The two
directories under `messages/profiles/` are ownership layers, not selectable
site profiles. Do not add runtime profile selection back around them.

## Loading rules

- Server runtime loads messages through `src/lib/i18n/load-messages.ts` and
  `src/i18n/request.ts`.
- Never self-fetch the app's own message JSON from server runtime.
- Root layout must emit the correct server-rendered `<html lang={locale}>`.
- Call `setRequestLocale(locale)` in **every layout and every page** under
  `[locale]/`, not only the root layout. next-intl requires this call in each
  layout and page for static locale rendering; under Next.js 16, missing it in a
  single page turns the whole site dynamic.
- Runtime boundary files such as `src/middleware.ts` use the edge-safe routing
  config from `src/i18n/routing-config.ts`.
- Do not import `src/i18n/routing.ts` from middleware or other edge-sensitive
  entrypoints; it is for app/page usage.

## Documented exemptions

Two files legitimately break the "every page calls `setRequestLocale`" and
"all buyer-visible text via translation keys" rules. These are framework
limitations, not oversights — do not "fix" them by wiring next-intl in.

- `src/app/global-error.tsx` — the global error boundary renders its own
  `<html>`/`<body>` and runs **outside** the `[locale]` layout tree, so the
  next-intl request context and provider are not available here. Its UI strings
  live in a local `translations` object with a fixed `en` `<html lang>`. This is
  the last-resort boundary that must render even when locale setup itself failed.
- `src/app/[locale]/[...rest]/page.tsx` — the catch-all only calls `notFound()`
  to route unmatched paths into the localized `not-found.tsx`. It renders no
  buyer-visible text and intentionally omits `setRequestLocale`: it never
  produces localized output, and `generateStaticParams` returns a single
  `__not-found-placeholder` entry that `notFound()`s during pre-render. Adding
  `setRequestLocale` here would imply a localized render that never happens.

## Usage

| Need | API |
| --- | --- |
| Server Component text | `getTranslations()` |
| Metadata with known locale | `getTranslations({ locale, namespace })` |
| Client Component text | `useTranslations()` |
| Rich text | `t.rich()` |
| Cached/helper code | pass explicit `locale` |

Inside `"use cache"` helpers, avoid request-bound APIs such as
`getTranslations()` without arguments or `getLocale()`. Pass `locale`.

In metadata helpers, pass the known locale explicitly. Do not rely on request
locale discovery inside cached or prerender-sensitive helpers.

## Cache Components + next-intl anchors

- next-intl's `cacheComponents` support is still an unfinished upstream open
  issue. Do not render the `Link` from `@/i18n/routing` inside a `"use cache"`
  boundary. Even passing an explicit `locale` triggers a `headers()` call inside
  the boundary and throws. Instead, create the `Link` outside the cached
  boundary and pass it in as `children`.
- `setRequestLocale` is expected to be replaced by `next/root-params` upstream.
  It is not deprecated yet and remains the officially recommended API today;
  this is a tracking note, not a migration instruction.

## Key rules

- Key segments use camelCase.
- Namespace keys cannot contain `.`.
- Normal depth is 2-4 levels.
- Add keys to every configured locale in `LOCALES_CONFIG`.
- After changing a physical pack, run `pnpm content:check`.

Run translation validation after message changes:

```bash
pnpm content:check
```

Use `loadCompleteMessages(locale)` only when a full catalog is truly needed.

Client payload is controlled by the namespace pick in
`src/lib/i18n/client-messages.ts` (`pickClientMessages` / `CLIENT_MESSAGE_NAMESPACES`).
All physical packs are loaded and merged before the namespace pick runs. To
shrink the client payload, narrow the namespace pick — not the pack layout.
