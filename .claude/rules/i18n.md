# i18n Rules

Use this file when changing translation keys, message loading, locale routing,
or next-intl usage.

For page prose and FAQ ownership, use `content.md`.

## Runtime truth

Runtime canonical source is physical message packs:

```text
messages/base/{locale}/{critical,deferred}.json
messages/profiles/{profile}/{locale}/{critical,deferred}.json
```

Generated compatibility files may exist at `messages/{locale}/{critical,deferred}.json`,
but authoring changes belong in the base/profile packs.
Examples: `messages/en/critical.json` and `messages/en/deferred.json` are
compatibility outputs, not primary authoring sources.

Runtime loads composed pack imports through `src/lib/i18n/message-pack-loader.ts`.
Runtime loads bundled split imports through `src/lib/i18n/load-messages.ts`.
Do not treat `public/messages/**` as an authoring or proof surface.

Current repo truth does **not** include a live `src/sites/**/messages/**` runtime overlay layout.

## Loading rules

- Server runtime loads split bundles through `src/lib/i18n/load-messages.ts` and
  `src/i18n/request.ts`.
- Never self-fetch the app's own message JSON from server runtime.
- Root layout must emit the correct server-rendered `<html lang={locale}>`.
- In static locale rendering, keep `setRequestLocale(locale)` in
  `[locale]/layout.tsx`.
- Runtime boundary files such as `src/middleware.ts` use the edge-safe routing
  config from `src/i18n/routing-config.ts`.
- Do not import `src/i18n/routing.ts` from middleware or other edge-sensitive
  entrypoints; it is for app/page usage.

## Usage

| Need | API |
| --- | --- |
| Server Component text | `getTranslations()` |
| Metadata with known locale | `getTranslations({ locale, namespace })` |
| Client Component text | `useTranslations()` |
| Rich text | `t.rich()` |
| Cached/helper code | pass explicit `locale` |

Inside `"use cache"` helpers, avoid request-bound APIs such as
`getTranslations()` without arguments or `getLocale()`. Pass `locale`.

In metadata helpers, pass the known locale explicitly. Do not rely on request
locale discovery inside cached or prerender-sensitive helpers.

## Key rules

- Key segments use camelCase.
- Namespace keys cannot contain `.`.
- Normal depth is 2-4 levels.
- Add keys to every configured locale in `LOCALES_CONFIG`.
- If split source changes, run `pnpm content:check`.

Run translation validation after message changes:

```bash
pnpm content:check
```

Use `loadCompleteMessages(locale)` only when a full catalog is truly needed.
Prefer critical/deferred or namespace-level access for client payload control.
