# Guideline: Digger 调研助手指南

**文档性质**：流程型 guideline——指导主会话如何使用 em-digger subagent 完成调研任务
**适用范围**：主会话与用户对话中遇到需要调研的问题
**核心问题**：如何判断调研模式、编写高质量 prompt、处理返回结果？

---

## 核心定位

**Digger 是什么**：
- 调研助手，不是独立任务
- 主会话的"外派员工"，去收集信息后回来汇报
- 不占用主会话 token，保护对话上下文

**使用场景**：
- 用户主动要求调研（"帮我查一下"、"研究一下"）
- 对话中遇到需要验证的信息
- 需要对比多个方案/工具/方法

**不适用**：
- 简单的事实性问题（主会话直接回答）
- 用户没有调研意图的闲聊
- 代码实现类任务

### 设计思路

**为什么分 4 种模式**：用户说「查一下」和「全面调研」期望完全不同。模式不是复杂度分级，而是**用户期望管理**——quick 用户期望即时答案，exhaustive 用户期望深度报告。错配期望会导致：quick 用户觉得等太久，exhaustive 用户觉得不够深入。

**为什么 quick 不生成文档**：quick 的价值在于「快」。生成文档需要结构化整理、写入文件，会让用户等待，破坏即时体验。quick 模式的本质是「代替用户搜几下」，而非「产出调研成果」。

**为什么用「主会话外派员工」比喻**：强调 Digger 的**从属关系**——它不是独立任务，而是主会话的延伸。调研完成后，结果要回到主会话继续对话，而非调研完就结束。这个比喻帮助 AI 理解：不要在调研中做主会话该做的决策。

### 术语：Hop（跳转）

**Hop** 是调研深度的计量单位，指从一个信息源跳转到下一个信息源的过程。

```
1 Hop：搜索 → 获取结果（可能多次搜索，但都是第一层）
2 Hop：从第一层结果中发现线索 → 追踪深入搜索
3 Hop：继续根据新发现深入追踪
...
```

**Hop vs 搜索次数**：Hop 衡量深度（层级），搜索次数衡量广度。1 Hop 可能包含多次搜索（同一层级内横向扩展）。

**Multi-Hop 调研**：不满足于第一层搜索结果，而是从结果中提取关键词、引用来源、相关概念，层层深入。Hop 数越多，调研越深入，但也越耗时。

## 模式速查

| 用户说... | 选择模式 |
|-----------|----------|
| "查一下"、"怎么配置"、"有没有方法" | quick |
| "了解一下"、"介绍一下" | standard |
| "深入研究"、"对比一下"、"最佳实践" | deep |
| "全面调研"、"彻底研究"、重要决策前 | exhaustive |

> **判断优先级**：用户明确说出模式关键词时直接采用；否则根据问题范围判断——范围小且目标明确用 quick，一般性了解用 standard，需要对比或综合用 deep，重要决策用 exhaustive。

> **时效敏感领域注意**：涉及 AI 工具（NotebookLM/Claude/ChatGPT/Cursor 等）、"是否支持"、"能不能"等产品功能类问题时，em-digger 会自动执行时效敏感性检查（官方源优先、否定性结论标注 `[⚠️ 建议实测验证]`）。主会话无需额外处理，但应知悉这类问题的调研结论可能需要用户实测验证。

## 四种模式

| 模式 | 适用场景 | Hop 数 | 输出 |
|------|---------|--------|------|
| **quick** | 查配置、查方法、快速验证 | 1 | 直接返回文本 |
| **standard** | 一般性调研、入门了解 | 2-3 | 生成文档 |
| **deep** | 复杂问题、对比分析、方案选型 | 3-4 | 生成文档 |
| **exhaustive** | 重要决策、全面调研 | 5 | 生成文档 |

### Quick 模式

**识别信号**：
- "帮我查一下"、"xxx 怎么配置"
- "有没有 xxx 的方法"
- 问题范围小、目标明确

**特征**：
- 2-3 次搜索
- **不生成文档**
- 直接返回文本

### Standard 模式

**识别信号**：
- "了解一下"、"介绍一下"
- 一般性知识获取
- 不需要深度对比

**特征**：
- 4-6 次搜索，2-3 Hop 深度
- **生成文档**

### Deep 模式

**识别信号**：
- "深入研究一下"、"系统了解一下"
- "对比一下 A 和 B"
- "最佳实践是什么"
- 问题范围大、需要综合

**特征**：
- 6-10 次搜索，3-4 Hop 深度
- **生成文档**

### Exhaustive 模式

**识别信号**：
- "全面调研"、"彻底研究"
- 重要决策前的信息收集
- 用户明确要求最大程度调研

**特征**：
- 10+ 次搜索，5 Hop 深度
- **生成文档**

**返回处理**：
- quick：主会话直接展示结果，继续对话
- 其他：主会话读取文档，展示摘要，基于调研结果继续对话

## Prompt 编写规范

### Quick 模式 Prompt

```
调研任务：{用户的问题}
模式：quick
```

**示例**：
```
调研任务：VSCode 如何配置 Python 虚拟环境自动激活
模式：quick
```

### Standard 模式 Prompt

```
调研任务：{用户的问题}
模式：standard

背景：{简要上下文}
输出文件：@/docs/digger.{主题}.md
```

**示例**：
```
调研任务：Obsidian 的 Dataview 插件基础用法
模式：standard

背景：用户想用 Dataview 做笔记查询
输出文件：@/docs/digger.obsidian-dataview.md
```

### Deep 模式 Prompt

```
调研任务：{用户的问题}
模式：deep

背景：{从对话中提取的上下文}
关注点：
- {用户特别关心的方面 1}
- {用户特别关心的方面 2}
输出文件：@/docs/digger.{主题}.md
```

**示例**：
```
调研任务：PKM（个人知识管理）的主流方法论对比
模式：deep

背景：用户正在整理个人笔记体系，使用 Obsidian
关注点：
- 卢曼卡片盒、PARA、MOC 等方法的核心理念
- 各方法的优缺点对比
- 与 Obsidian 的适配性
输出文件：@/docs/digger.pkm-methodologies.md
```

### Exhaustive 模式 Prompt

```
调研任务：{用户的问题}
模式：exhaustive

背景：{详细上下文}
关注点：
- {关注方面 1}
- {关注方面 2}
- {关注方面 3}
约束：{时间范围、技术栈限制等}
输出文件：@/docs/digger.{主题}.md
```

**示例**：
```
调研任务：当前年份 AI 编程助手工具全面对比
模式：exhaustive

背景：用户需要为团队选型 AI 编程工具
关注点：
- 各工具的核心能力对比（代码补全、对话、重构）
- 定价模式和企业版功能
- 隐私和安全考量
- 与主流 IDE 的集成度
约束：只考虑有企业版的工具
输出文件：@/docs/digger.ai-coding-tools-{year}.md
```

> **注意**：时效性调研中的年份由 em-digger 自动获取（`date +%Y`），无需在 prompt 中硬编码。

### 检查清单

派发调研前，用以下问题自检：

| # | 检查项 | 验证要点 | 适用模式 |
|---|--------|---------|---------|
| 1 | 任务描述清晰吗？ | ① 能否一句话说清"要调研什么"？② 有没有模糊表述（"查一下相关的"、"了解一下情况"）？ | 全部 |
| 2 | 模式选对了吗？ | ① 用户期望是"快速答案"还是"深度报告"？② 问题范围是否与模式匹配？（小范围→quick，大范围→deep） | 全部 |
| 3 | 背景信息够吗？ | ① 对话中的上下文有没有带入 prompt？② 用户特别关心的方面有没有列出？ | deep/exhaustive |
| 4 | 输出路径对吗？ | ① 文件名是否简洁、能一眼看懂？② 路径格式是否符合 `@/docs/digger.{主题}.md`？ | 非 quick |
| 5 | 需要确认吗？ | ① 是否已展示搜索方向让用户确认？② 用户有没有说"直接调研"跳过确认？ | deep/exhaustive |

## 计划确认（deep/exhaustive 模式）

**为什么需要确认**：deep/exhaustive 模式消耗较多 token 和时间，用户应有机会：
- 确认搜索方向是否符合预期
- 补充遗漏的关注点
- 调整调研范围

**确认时机**：prompt 编写完成后、调用 em-digger 前

**确认内容**：
- 搜索方向（从用户输入分解出的子问题）
- 关注点（用户特别在意的方面）
- 预计产出（文件路径、搜索深度）

**用户响应**：
- 确认 → 执行
- 调整 → 修改 prompt 后重新确认
- 取消 → 结束

**跳过条件**：
- 用户明确说「直接调研」「不用确认」
- quick/standard 模式（无需确认）

## 输出文件命名

**格式**：`@/docs/digger.{主题}.md`

**主题命名规则**：
- 英文 kebab-case
- 简洁表达核心内容
- 不加日期（除非是时效性调研）

**示例**：
```
@/docs/digger.pkm-methodologies.md
@/docs/digger.vscode-remote-dev.md
@/docs/digger.nextjs-vs-remix.md
@/docs/digger.ai-coding-tools-{year}.md  # 时效性调研加当前年份
```

---

## 文档结构

文档结构由 em-digger subagent 定义，主会话不需要关心具体格式。

**主会话只需知道**：
- 文档包含 Executive Summary（用于快速展示给用户）
- 文档包含置信度评估
- 文档包含来源汇总

---

## 多轮调研

**场景**：同一对话中多次调用 digger

**处理方式**：
- 每次调用都是独立的 subagent
- 主会话负责串联各次调研结果
- deep 模式的文档可以在后续调研中引用

**示例流程**：
```
用户：帮我查一下 Obsidian 的 dataview 插件怎么用
    → quick 模式，返回基础用法

用户：深入研究一下 dataview 的高级查询
    → deep 模式，生成 @/docs/digger.obsidian-dataview.md

用户：再对比一下 dataview 和 db-folder
    → deep 模式，生成 @/docs/digger.obsidian-dataview-vs-dbfolder.md
    → 主会话综合两份文档给出建议
```

## 经验总结

| 教训 | 说明 |
|------|------|
| **模式判断优先** | 先判断 quick/standard/deep/exhaustive，再编写 prompt |
| **背景信息关键** | 非 quick 模式的背景和关注点决定调研质量 |
| **文件命名简洁** | 主题名要一眼看懂，不要过长 |
| **结果要用起来** | 非 quick 模式返回后主会话要读取文档，不是生成就结束 |
| **时效敏感需实测** | AI 工具/产品功能类问题，调研结论可能需用户实测验证（em-digger 会标注 `[⚠️ 建议实测验证]`） |
