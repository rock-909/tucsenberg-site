# 代码库审计摘要 — 2026-05-09

审计模式：`code`

## 1. 一句话结论

这个仓库整体比典型 AI 堆出来的项目健康很多，基础检查是干净的；本轮最该担心的不是“跑不起来”，而是联系表单里有两处会影响买家询盘质量和失败提示的细节。

## 2. 四栏判断

| 维度 | 等级 | 人话说明 |
|---|---|---|
| Code health | Ok | 结构、类型、依赖边界都比较稳，没有看到会立刻阻断上线的代码级灾难。 |
| Proof health | Not assessed | 本轮是 code 模式，只顺手看了部分测试证据，没有完整审测试/CI 可信度。 |
| Truth-source health | Not assessed | 本轮没有完整审文档、规则、内容和真实业务配置是否漂移。 |
| Repairability | Strong | 发现的问题都很集中，不需要大拆项目，适合用小范围修复收口。 |

## 3. 现在最重要的 3 个问题

- **买家填写的“主题”会被系统静默改写**：客户写的原始主题不会原样进入后续线索系统，而是被英文关键词猜成几个固定类别。
- **表单填错时，系统知道哪里错了，但没有告诉用户**：后端已经算出了字段级错误，API 返回时把这些细节丢掉了。
- **安全扫描覆盖有一个工具缺口**：`semgrep` 这台环境里不可用，所以不能说安全扫描器覆盖完整。

## 4. 这些问题对业务的实际影响

- 对询盘：销售看到的主题可能不是客户真实表达，尤其是中文主题、行业术语、具体项目名，容易被归成“其他”。
- 对转化：用户填错表单时只看到笼统失败，可能不知道怎么改，增加放弃提交的概率。
- 对审计可信度：代码基础检查是干净的，但安全扫描工具链还缺一块，后续做 release / security review 时要补上。

## 5. 建议动作顺序

| 优先级 | 动作 | 为什么现在做 |
|---|---|---|
| P0 | 保留联系表单 `subject` 原文；如需分类，新增单独分类字段 | 这是最直接影响销售判断的真实线索数据。 |
| P1 | 让 `/api/contact` 把字段级错误 `details` 返回给前端 | 已经有错误细节和前端展示能力，只是中间被截断，修复收益高。 |
| P2 | 补齐 `semgrep` 或明确替代安全扫描命令 | 避免以后说“安全扫描完整”但实际漏了一类工具。 |

## 6. 这轮审计没有否定什么

- 没有看到联系表单在代码层面存在“先发邮件再入库”的顺序问题；当前是 Airtable 成功后再发 owner email，确认邮件是非阻塞的。
- 没有看到明显的 secret 泄露、Next.js Server/Client 边界乱用、i18n 路由直接坏掉这类高危问题。
- 没有把测试质量、CI、Cloudflare preview/deploy、文档真相漂移当成已审通过；这些不属于本轮 code 模式完整范围。

## 7. 下一步怎么接着做

如果继续修，建议先只修两件事：第一，保留买家填写的主题原文；第二，打通字段级错误提示。修完后跑 `pnpm type-check`、`pnpm lint:check`，再补一条 `/api/contact` 的路由测试证明错误详情能返回到前端。
