## S1 — 产品真相收口：单一数据源（P1，结构债根因）

### 问题

同一个产品的规格现在有两套"看起来都权威"的数据：

- 页面正文真相：`src/constants/tucsenberg-product-pages.ts`（`getTucsenbergProductPage()`，
  页面 sections 由它渲染，`src/app/[locale]/products/[market]/page.tsx:181-182,230+`）
- JSON-LD / 旧规格路径：`src/constants/product-specs/tucsenberg-product-lines.ts`
  （`getMarketSpecsBySlug()` → `market-page-data.ts` → `market-jsonld.ts`，
  `page.tsx:225-228` 输出 JSON-LD）

已发生的实际后果：首轮的 "TODO-OWNER" 只出现在其中一套；图片字段写着不存在的
`/images/products/tb-*-placeholder.svg`（五个文件都不存在），靠 guard 兜住而非数据干净。
**后续每次改产品参数都要"记得改两处"，JSON-LD 随时可能和买家看到的页面不一致。**

### 修法（采用方案 1，已由审计和 ponytail 视角双重确认）

**`TUCSENBERG_PRODUCT_PAGES` 是当前站的产品 runtime truth。**

1. JSON-LD 改为从 `TucsenbergProductPage` 生成：重写 `market-jsonld.ts` 的数据来源，
   `market-page-data.ts` 不再为当前产品详情页提供 spec truth。
2. 图片字段改为显式状态而不是"不存在的路径"：
   `image: { status: "real" | "pending" | "omitted", src?: string }`（或等价的可判别联合）。
   `pending`/`omitted` 时 JSON-LD 省略 image 字段、页面走已批准的 fallback。
   **禁止**再用一个 404 路径表示"待定"。
3. `product-specs/tucsenberg-product-lines.ts` 降级为 legacy/starter fixture：
   文件头注释标明"非当前站产品真相"，从当前产品页数据流中断开。
4. 加 contract 测试锁住单一真相：
   - 产品详情页与 JSON-LD 的规格来自同一来源（可对比关键字段一致性）；
   - `SITE_CONFIG` 不得出现 `brandAssets` 访问（架构测试层面禁掉这类越权读取）；
   - 产品 image 字段若为 `real` 必须对应 `public/` 下真实存在的文件
     （消除"数据层写死链"这一类 bug 的结构条件）。
5. 更新 `docs/项目基础/替换边界.md` / 相关 truth docs 中产品数据 owner 的描述。

## S4 — ESLint 配置治理（P2，独立）

### 问题

`eslint.config.mjs`（1084 行）规则层互相打架：
- `no-magic-numbers` 在同一 rules object 重复定义（:373-404 与 :459-470+）；
- 先给 scripts 渐进 warning（:699-708），最后 `scripts-directory-overrides`（:979-1041）
  又把 complexity / max-lines / max-statements / max-params 全关。

后果：最需要结构治理的脚本区（`starter-checks.js` 1207 行、
`content-readiness.js` 994 行、`content-slugs.js` 762 行）反而绕开了约束，
质量门"看起来严格、实际失真"。

### 修法

1. 提取共享 `MAGIC_NUMBER_IGNORE_LIST`，删除重复定义。
2. 删除或收窄末尾的 `scripts-directory-overrides`；scripts 保留 **warning 级**
   max-lines / max-statements / complexity（不是立刻 error，口径诚实即可）。
3. `starter-checks.js` 做**搬迁型重构**：CLI routing 留在 `starter-checks.js` 作 facade，
   `content-manifest`、`component-governance` 等实现搬到 `scripts/quality/checks/*`，
   保持 backward-compatible exports（现有调用方式一律不变）。
4. 阈值口径对齐 `.claude/rules/code-quality.md:22-26`。

### 验证

```bash

node scripts/starter-checks.js component-governance
```

## S5 — lead pipeline 并行化（P3，独立小改）

### 问题

`processContact()` / `processProduct()` 把 owner email 与 Airtable 写入**串行** await
（`src/lib/lead-pipeline/process-lead.ts:186-207,286-305`），
但两者无数据依赖、各自已 catch 返回 boolean，策略语义是
`LEAD_DELIVERY_POLICY = "email-first-storage-optional"`（任一通道成功即成功）。
串行只是白白拉长买家提交等待。

### 修法

```ts
const [emailSent, recordCreated] = await Promise.all([
  sendProductOwnerEmail(lead, context),
  createProductLeadRecord(lead, context),
]);
```
contact 同理。confirmation email 仍在 contact 成功后 schedule。
**必须用测试锁住语义**：任一通道成功即成功；两者全败才失败；
核对日志顺序与错误聚合不回归（canary 依赖 `recordCreated` / `ownerNotified` 两个记录位）。

### 验证

