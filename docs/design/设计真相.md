# Design Truth

Tucsenberg current site design truth.

这份文档记录 Tucsenberg 当前英文站的设计真相。它会沿用有用的
starter 治理规则，但不是通用 starter 说明，也不是最终品牌规范。

Current boundary:

- 当前公开站点是 English-only，没有 `/zh` 设计入口。
- 当前运行态组件、tokens、内容和 proof 优先于旧 starter 记录。
- starter 设计材料只解释继承来源；不能覆盖 Tucsenberg 当前页面真相。

## 当前设计定位

Tucsenberg 当前沿用的 inherited starter design baseline 是：

- 清晰
- 可信
- 现代
- 克制
- 易替换
- 便于 AI 复用和维护

当前站点更具体的方向是 **Modern Technical B2B**，也可以描述为
**Product-led Industrial B2B**。

含义：

- 比传统外贸站更年轻、更国际化；
- 比 SaaS 或 AI 官网更实在、更可信；
- 信息结构服务产品体系、应用场景、规格标准、供应能力和询盘转化。

默认避免两种偏差：

1. 过度装饰，导致内容和行动路径不清楚。
2. 组件随手新增，导致后续 AI 难以复用和维护。

## 当前品牌表达

当前视觉是 Tucsenberg 从 starter 继承的可替换基线，不是最终品牌。

设计上要做到：

- 当前业务内容看起来完整可运行；
- 后续品牌微调时不需要逐个组件改颜色；
- 组件状态、表单、卡片、导航等有统一规则；
- Storybook 可以作为组件预览和审核面。

## 当前视觉基线

现有 token、圆角、阴影、网格和布局节奏可以继续作为当前控制面板。  
但派生项目确认真实品牌后，可以替换 token 值和视觉风格。

### 色彩

- Current truth: Tucsenberg uses the inherited replaceable role-based color system.
- Stable interface: semantic roles and token architecture.
- Provisional value: current blue/neutral palette.

规则：

- 生产组件不直接写品牌 hex。
- 改品牌色先改 token。
- 如果 token 角色变化，需要同步 `docs/design/色彩系统.md` 和相关测试。

### 字体

- 当前主字体基线：`src/app/globals.css` 里的 system stack +
  中英文混排回退策略。
- 排版优先清晰、稳定、可读。

### 形状与阴影

- 默认圆角和阴影保持克制。
- 阴影用于区分层级或表达交互，不用于填充空白。

### 动效

- 当前 motion governance 真相源是 `docs/design/动效治理.md`。
- 动效默认服务状态、层级和路径，不服务装饰。
- 常规状态变化保持短促；页面级 reveal、page transition、长动效或新增
  motion dependency 需要单独 proof。
- 首屏主张内容必须默认可见，不靠客户端动画出现。
- `prefers-reduced-motion` 是硬要求，不是可选增强。

## 当前 Vercel-inspired calibration

这项 calibration 是 documentation-first，not a visual migration。

规则：

- Calibration remains documentation-first until a local visual proof is approved.
- Calibration is not a visual migration by itself.
- 吸收 Vercel 的清晰层级、克制留白、组件状态、排版纪律和短促动效。
- 不复制 Vercel 的黑白开发者平台身份。
- Do not adopt Geist as the default site font.
- Do not copy Vercel token names into the project contract.
- Do not bundle Apple proprietary font files.
- Apple-first system typography 只作为评审语言：当前仍使用跨平台 system
  stack，在 Apple 设备上自然使用 Apple 系统字；评审重点是清晰、稳定、可读。
- 没有单独 visual proof 前，不打包 Apple 字体，不改 runtime CSS、全局 token、
  `--radius`、字体加载、控件尺寸或页面布局。
- Do not silently change global `--radius`, runtime font loading, control
  sizing, or token values without separate visual proof.

## Runtime Button pilot

The runtime typography, spacing, and radius pilot starts with Button only.

Allowed in this pilot:

- Apple-first system typography language while keeping the cross-platform
  system stack;
- Button component tokens for height and radius;
- Button wrapper tests and real-page computed-style proof.

Not allowed in this pilot:

- font file loading;
- Geist adoption;
- global `--radius` migration;
- Card, Input, or section radius migration;
- page layout rewrites;
- color-system migration;
- new dependencies.

## 当前 B2B evidence 规则

重要主张旁边要有证据。证据可以是产品族、应用场景、规格、材料、
工艺、标准、产能、质量能力、认证范围、案例约束或询盘下一步。

默认避免两种偏差：

1. Alibaba-style density：产品瓦片、徽章、供应商信息和 trust icons 抢注意力。
2. SaaS-style emptiness：抽象 slogan、通用卡片和漂亮 surface 缺少产品事实。

## Inherited starter visual adaptation boundary

This section remains useful for inherited starter surfaces and future derivation
work. It is not a license to replace Tucsenberg current page truth with demo
company details or Vercel's identity.

Preserve:

- clear product families or product-system / service-system hierarchy;
- application fit close to the claim it supports;
- verified proof near buyer-risk statements;
- a visible inquiry path;
- restrained surface hierarchy and readable typography.

Replace:

- demo company facts;
- example product names;
- placeholder proof;
- generic resource titles;
- contact and response expectations.

Avoid:

- Vercel developer-platform motifs such as deploy, console, terminal, or
  platform dashboard language unless the real derived business is actually a
  developer platform;
- unverifiable customer counts, badge walls, and card walls;
- traditional export-site density dumps;
- SaaS-style empty whitespace without operational detail.

Do not rely on fake proof, unverifiable claims, badge walls, or card walls to
manufacture trust. Derived projects must not imitate Vercel's developer-platform
identity.

## 当前 spacing 与 motion 评审基准

Spacing 只作为评审基准；没有单独页面级 visual proof 前，不改 runtime spacing：

- 8px：紧密内部组；
- 16px：相关内容组；
- 24px：卡片 padding 和常见内容块；
- 32px 到 40px：较大的 section 内部分组；
- 56px mobile / 72px desktop：现有页面 section rhythm，除非后续页面级 proof 证明需要调整。

Motion 的完整治理规则看 `docs/design/动效治理.md`。没有单独 visual proof 前，
不加全站动画：

- 常规状态变化约 150ms；
- popover、menu、小 panel 约 200ms；
- 优先 CSS 和已有 state attribute，不用 JavaScript orchestration；
- 页面级 reveal、page transition、超过 300ms 的动效或新增依赖需要单独
  proof；
- Do not animate layout properties by default.
- Do not gate content visibility on animation.
- 尊重 `prefers-reduced-motion`。

## 当前交互与动效原则

- 动画服务于理解，不服务于表演。
- 默认使用 restraint + speed + purposeful motion。
- 常规交互动效应保持短促。
- 尊重 reduced motion。

## 当前组件治理原则

- 先复用，再新增。
- 低层 UI 组件不能塞业务文案。
- 可复用组件要有清晰 variants。
- 视觉上容易漂移的组件应补 Storybook story。
- 新组件如果只是现有组件的轻微变体，优先扩展现有组件。

## 当前 UI 底座决策

当前采用本地 UI wrappers + Radix Primitives，详见
`docs/决策记录/UI基础方案.md`。

- Radix Primitives 是复杂交互默认底座。
- Radix-style 1-12 色阶是长期颜色纪律。
- Native HTML、Tailwind 和项目 tokens 负责普通控件、页面布局、响应式结构和品牌表达。
- `src/app/globals.css` 里的项目 tokens 继续是运行时颜色真相源。
- `@radix-ui/themes` 已退役，生产 UI 禁止重新引入。

判断 UI 走哪条路时，不按页面区域粗分，而按行为分：

- 有交互、状态、校验、focus、loading、error、selected 等控制逻辑的 UI，
  先判断原生 HTML 是否足够；需要复杂键盘、焦点、浮层或选择行为时进入
  Radix Primitive wrapper。
- 纯叙事、品牌图、页面排版、静态内容层级，继续用 Tailwind + 项目 tokens。

Hero 里的 CTA 按钮可以用项目 Button wrapper，因为它是控件。Hero layout
本身不应交给 vendor UI 系统。

## 当前不采用的方向

- 把 starter 做成某个具体行业的固定品牌。
- 每个页面随手新建一套按钮、卡片、表单。
- 大量硬编码颜色。
- 用 fake proof、fake logo、fake 客户来制造信任。
- 重新引入 Radix Themes 或任何全局 vendor theme runtime。
- 用 vendor UI 系统接管 Hero、产品故事、工厂证明、Footer 或页面叙事结构。
