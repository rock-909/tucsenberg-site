# Stitch 使用指南

面向 Tucsenberg 洪水挡板网站的 Stitch AI 设计工具使用说明。**这份文档主要写给网站主(非技术）看**，讲清楚它是什么、怎么用、该给它什么信息、不该给什么。最后附一段「连不上怎么办」。

---

## 一分钟看懂

- **Stitch 是什么**：Google 的 AI 界面设计工具，背后是 Gemini 3 模型。你用**一句话描述**想要的页面，它就生成出可视化的设计稿（网页样式 + 截图）。
- **怎么触发**：它已经接进了 Claude Code。**你不用碰任何按钮或参数，直接用中文跟 Claude 说需求就行**，Claude 替你操作。
- **两条铁律，先记住**：
  1. **它只出设计草稿，不出上线代码。** 稿子定了以后，由 Claude 按本项目的组件规范去实现真正的网页——Stitch 的代码不能直接上线（原因见下文「定稿之后」）。
  2. **你打给它的每个字都会上传到 Google 云。** 所以涉密信息不能给（详见「不该给什么信息」）。

---

## 你怎么用它

不需要学任何操作，就是跟 Claude 对话。一个典型流程长这样：

```
你：用 stitch 建个项目，做一个洪水挡板的产品落地页，面向海外采购商，要有规格参数表和询盘按钮
Claude：（建项目 + 生成设计，等 1~2 分钟）好了，你看这版
你：不错，但首屏再大气点，另外给我 3 个不同配色的版本
Claude：（生成修改版和变体）
你：第 2 个配色好，下载到本地
Claude：（存到项目里）→ 之后我参考它去改真实网站
```

> ⏳ **生成设计比较慢，一次一两分钟是正常的**，不要以为卡住了。

---

## 该给 Stitch 什么信息

给得越具体，设计越贴。建议每次说清这几样：

| 要说的 | 举例（洪水挡板场景） |
| --- | --- |
| **页面目的** | 「产品落地页，核心目标是让海外采购商提交询盘」 |
| **受众是谁** | 「面向欧美 B2B 采购经理、经销商、OEM 客户，不是普通消费者」 |
| **页面要有哪些板块** | 「首屏卖点 → 规格参数表 → 应用场景 → 认证资质 → 询盘表单」 |
| **视觉风格倾向** | 「工业、可靠、专业，冷色调为主，不要花哨」 |
| **手机还是桌面** | 「按桌面端设计」（Claude 会设好设备类型） |
| **已打算公开的产品信息** | 规格字段（防水高度、材质、拼装方式）、OEM/批发卖点、认证类型——**这些本来就要放官网，给没问题** |

**一句话对比：**

> ❌ 差：「做个产品页」——太空，它只能瞎猜
>
> ✅ 好：「桌面端 B2B 产品落地页，面向欧美采购商；首屏突出防水性能、材质、快速拼装三个卖点，下面接规格参数表、应用场景图、认证 logo 墙，底部放询盘表单；工业冷色调、专业可靠、不要花哨」

---

## 不该给什么信息

### 第一组：会泄密的（上传即等于交给 Google，严禁）

- 真实**客户名单、询盘邮箱、联系人**
- **内部价格、成本底线、利润率、报价策略**
- 未公开的**商业计划、供应商信息、独家工艺细节**
- 任何**密钥、密码、后台地址**（比如 Stitch 的 API key、Turnstile 密钥）
- 员工或合作方的**个人隐私信息**

> **判断法则：这句话能不能直接印在公开官网首页上？不能，就别给 Stitch。**

### 第二组：给了没用、甚至帮倒忙的

- **技术栈细节**——Next.js、Radix、Tailwind、多语言这些**别跟 Stitch 说**。它只出视觉稿，不写你的生产代码，说了用不上还干扰它。
- **大段正式文案**——早期探索用示意文字就行，版式定了再灌真文案，省时间。
- **像素级细抠**——「这里间距改 8px」这种没必要在 Stitch 阶段说，那是 Claude 后面落地时的活。

---

## 它能做的事（四类）

你用大白话说，Claude 会挑对应的能力：

1. **生成设计**——从一句话描述生成整个页面；也能新建项目当容器。
2. **迭代和微调**——对现有页面文字修改；一次生成 1~5 个变体，还能指定**只变**布局／配色／图片／字体／文案，幅度可选小改／平衡／大改重构。
3. **拿走成果**——把设计下载到本地（网页样式 + 图片），Claude 就能参考它改真实网站；也能单独取某页的代码或截图。
4. **统一风格（设计系统）**——定义一套统一的颜色／字体／形状规范，一键套用到所有页面，保证整站视觉一致。

---

## 定稿之后 = 交给 Claude 落地（重要边界）

Tucsenberg 网站有一套严格的前端治理，Stitch **不能绕过它**：

- Stitch 出的网页代码 **不能直接上线**。它是设计参考，用来定视觉方向。
- 定稿后，由 Claude 按项目的 **Radix 组件体系**、`docs/design/设计真相.md`、组件使用手册来实现，并通过 `pnpm component:check` 硬门禁。
- 动效要遵守 `docs/design/动效治理.md`——动画得服务于状态和层级，不是为了「显得高级」。

**正确心智**：Stitch 负责「长什么样」，Claude 负责「按规范把它变成真网站」。两边分工，别把 Stitch 的 HTML 直接塞进代码库。

---

## 连不上怎么办（排障速查）

Stitch 每次启动都要连 Google 服务器，国内直连会被墙，所以**必须走代理**。配置已经调好并写在项目根目录的 `.mcp.json` 里，一般不用动。真遇到连不上，按下面查：

**最常见原因：代理没开。**

- Stitch 依赖 **Surge 代理**，端口 `6152`。**Surge 关了，Stitch 必然连不上。** 先确认 Surge 开着。
- 换了别的代理软件、或代理端口变了 → 改 `.mcp.json` 里 `HTTPS_PROXY` / `HTTP_PROXY` 两行的端口号。

**在 Claude Code 里恢复连接**：输入 `/mcp`，选 stitch → `Reconnect`，看到 `Reconnected to stitch` 即成功。

**配置速查**（`.mcp.json` 的 stitch 段，四个关键项缺一不可）：

| 配置项 | 作用 |
| --- | --- |
| `args: ["-y", ...]` | `-y` 让后台安装不卡在交互确认 |
| `STITCH_API_KEY` | 认证（对应你的 Stitch 账号） |
| `HTTPS_PROXY` / `HTTP_PROXY` = `http://127.0.0.1:6152` | 指向 Surge 代理 |
| `NODE_USE_ENV_PROXY: "1"` | **关键**，不加这条上面的代理设置不生效 |

**安全提醒**：当前 API key 曾在协作聊天中出现过，建议适时在 Stitch 后台轮换一个新的，换上 `.mcp.json` 即可。`.mcp.json` 已被 `.gitignore` 忽略，不会提交进代码库，密钥不会外泄到版本历史。
