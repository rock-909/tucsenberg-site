# Contact Mobile Language and Form Design

## Goal

Make the public demo starter feel closer to a real launch-ready B2B website by tightening two high-friction areas:

- Header utility controls on desktop and mobile.
- Contact form fields and submission contract.

## Confirmed behavior

### Desktop header

- Desktop utility controls render in this order: `Contact` then current-language selector.
- The two controls remain visually grouped on the right side of the header.
- The language selector remains a compact menu button.

### Mobile navigation

- Mobile header renders the compact conversion layout: Logo, `Contact`, menu button.
- The mobile header contact action is visible below 840px and links to the localized contact route.
- The mobile menu keeps the primary page links first: Home, Products, Blog, About.
- The contact action appears as a clear CTA inside the mobile menu.
- The language switcher appears after the CTA as a utility row.
- Language options are collapsed by default.
- The collapsed row shows the utility label and the current language.
- Tapping the row expands the available language options.
- Language option labels use their own language names: `English`, `简体中文`.
- The JavaScript language switcher preserves the current path directly.
- The no-JS mobile language fallback preserves the current path only when the browser sends a same-origin `Referer`; without that header it lands on the selected locale root.

### Contact form

- Replace separate `firstName` and `lastName` visible fields with one required `fullName` field.
- `fullName` label:
  - English: `Full name`
  - Chinese: `姓名`
- `fullName` uses browser autocomplete `name`.
- Company name remains visible but becomes optional.
- Email, message, privacy acceptance, Turnstile, honeypot, rate limit, and idempotency behavior stay intact.
- The submission pipeline continues to send downstream `firstName` and `lastName` values by splitting `fullName` at the lead-processing boundary.
- The split is a legacy CRM/email compatibility adapter, not a perfect human-name parser. Single-part names keep the full value in `firstName` and leave `lastName` empty.

## Non-goals

- No new locale.
- No new contact endpoint.
- No change to Turnstile, rate limiting, or idempotency.
- No redesign of the whole contact page.
- No native select-based language picker.
- No special-case hiding of the mobile contact CTA on the contact page.

## Proof

The change must be proven with focused tests for:

- Header desktop utility order.
- Mobile header renders Logo, Contact CTA, and menu button as first-level controls.
- Mobile language options collapsed by default and expandable.
- Contact form renders `fullName`, does not render visible `firstName` or `lastName`, and treats company as optional.
- Contact validation accepts optional company and required full name.
- Submission still reaches the canonical lead pipeline with downstream first/last name compatibility.
- Generated type hygiene: `next-env.d.ts` is ignored and regenerated by `next typegen`, not edited or committed as source.
