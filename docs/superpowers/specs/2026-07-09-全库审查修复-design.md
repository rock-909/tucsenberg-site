# 全库可维护性审查修复 · 设计文档

- 日期：2026-07-09
- 分支起点：`main`
- 依据：`docs/技术难题/审查2026-07/` 下 13 份分批审查报告 + 1 份跨模块横向报告（共 14 份）
- 授权记录：owner 明确授权删除 starter / 多 profile / product-specs 机制，**覆盖 `CLAUDE.md` 与 `维护规则.md` 的退役证明门槛**（见下"治理决策"）。

## 背景

对 `src/` 全部业务代码与 `scripts/` 做了分 13 批 + 1 轮跨模块的可维护性审查，共 475 条问题（高 65 / 中 195 / 低 215）。核心结论：

- 全仓约 **10,500–11,500 行生产不可达代码**，占 src（39,835 行，不含测试/stories）的 **26–28%**；且成建制（第二套首页集群、starter 多 profile 机制、ui 零调用组件等）。
- 8 个影响线上行为的真 bug（询盘静默失败、询盘换行被销毁、导航进度条挂死、CORS 域名分叉、邮件字体栈失效、年份过期、CSP 状态码/PII 脱敏）。
- 3 个系统性根因：① starter→derived 收缩后多 profile/多 locale 机制未退役；② "消灭魔法数字"被机械执行成批量生成常量（含 4 处语义盲借用）；③ 组件/配置面先建、页面后接、接完不回收，tests/stories 为死代码续命。

## 治理决策（重要）

C 桶（starter / 多 profile / `product-specs/**`，约 5,500 行）在 `docs/项目基础/替换边界.md` 与 `维护规则.md` 中是**受保护的派生工厂机制**，退役本需"退役证明"。owner 在本次工作流中明确授权直接退役、覆盖该门槛。因此本工作流**必须同步更新** `替换边界.md`、`维护规则.md`、`CLAUDE.md` 中对应规则段落，使治理文档与代码保持一致——否则只是把"文档 vs 代码脱节"这一审查问题在别处再造一遍。此方向与 `CLAUDE.md` 已声明的 "It is not a generic starter anymore" 一致。

## 范围原则

- **全部【客观问题】彻底修**（真 bug + 意外死代码 + 明确的重复/不一致）——这是"接管全部问题"的实义。
- **【风格偏好】顺手改**：改到同一文件时一并清理，但不为纯主观偏好单独造 diff。
- 14 份审查报告随本 spec 一并入库，作为每个 PR 的依据。

## 执行结构：~10 个 PR，按依赖安全顺序

每个 PR = 一个"问题簇"，小到可独立审查、可独立回滚。合并一律等 owner 审批，不自动 merge。

### 顺序约束（不可乱）

- **PR1 必须最先**：解除 `LinkHref`/`Locale` 删除陷阱，否则 PR3–5 的删除会砸坏活代码。
- **PR8 必须在 PR3–7 之后**：死代码删完、常量消费面收缩后再清理常量，误删风险最低。
- PR6/PR7 是 C 桶退役，含治理文档更新。
- PR9/PR10 是行为契约重构，放最后，需回归测试护航。

## 执行纪律

- 每个 PR 走完整 superpowers 流程：plan → TDD（改行为的先写失败测试）→ 跑对应 verification → 开 PR。
- 单个 PR 若执行时 diff 失控（尤其 PR2/PR6），就地拆成 a/b 子 PR 并告知 owner；总数可能略超 10，但不破坏"小而可审"。
- 分支从 `main` 切，每个 PR 独立分支。
- 删除一律移入回收/归档（不用 `rm`），符合全局与项目习惯。
- **合并一律等 owner 审批。**
