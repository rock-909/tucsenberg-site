# 部署设置

本项目只保留 Cloudflare/OpenNext 部署链路。

starter 不保留任何旧项目域名、worker 名、zone 信息或 secret。

环境变量说明见 `docs/website/env 设置.md`。真实密钥不要入库；`.env.example` 只作为派生项目配置参考。

## Cloudflare 平台标识

系统判断“现在跑在 Cloudflare 上”时，主要看 `DEPLOYMENT_PLATFORM=cloudflare`。

`DEPLOY_TARGET=cloudflare` 只是为了兼容旧配置保留下来的写法。新项目优先配置 `DEPLOYMENT_PLATFORM`，不要只依赖旧变量。

## 必须替换

- Cloudflare 账号、Worker 名称、访问路由/绑定域名
- 正式域名
- Wrangler / CI 部署密钥：`CLOUDFLARE_ACCOUNT_ID` 和 `CLOUDFLARE_API_TOKEN`
- Turnstile site key 和 secret：site key 是前端公开 key，secret 是服务端密钥，不能公开
- 邮件服务配置
- 表单接收配置
- Next Server Actions 加密 key：`NEXT_SERVER_ACTIONS_ENCRYPTION_KEY`

`NEXT_SERVER_ACTIONS_ENCRYPTION_KEY` 是 Cloudflare / Next Server Actions 兼容相关配置。preview 要配置稳定值，production 也要配置稳定值。不要每次发布前临时乱填；如果这个值反复变化，Server Actions 相关加密/校验可能不稳定，发布验证会挡住。

## Cloudflare 图片交付策略

Cloudflare image delivery strategy 分成 starter baseline 和 customer upgrade lane。

starter baseline 默认做法：不启用 Cloudflare Image Optimization，也不要求 Cloudflare Images、Image Transformations、Polish、Mirage、R2 或自定义 image loader。

默认图片策略是：

- 替换 `public/images/**` 里的 starter 示例图片；
- 上传前把真实图片导出到合适尺寸和格式；
- 保留 `next/image` 的布局、lazy loading 和 `sizes` 价值；
- Cloudflare/OpenNext 构建保持可部署，不依赖图片产品或付费功能。

customer upgrade lane：如果真实项目图片很多，或者需要自动调整尺寸、自动转换格式、统一管理图片版本，再单独选择 Cloudflare Transformations 或 Cloudflare Images。

启用前必须确认：

- 当前域名已经在 Cloudflare zone 下；
- 客户接受对应 plan、quota 和 billing；
- 图片来源、origin 限制和缓存策略明确；
- 有已经部署出来、能实际打开的 Cloudflare 地址，可以证明买家看得到图片。

不要把 Cloudflare 图片优化写成 starter 默认能力。它是派生项目的可选升级，不是这个 starter 的默认上线前提。

## Cloudflare 访问数据看板

官方演示部署走 Cloudflare。给站点负责人/项目负责人看的流量看板，会在服务端读取真实 Cloudflare zone 的访问数据。

如果要启用 `/ops/traffic`，需要配置：

- `CLOUDFLARE_ZONE_ID`
- `CLOUDFLARE_ANALYTICS_API_TOKEN`
- `CLOUDFLARE_ANALYTICS_HOSTNAME`
- `OPS_DASHBOARD_ACCESS_KEY`

建议再加一层外部保护：

- 用 Cloudflare Access 访问保护来保护 `/ops/*`

不要把 API token（接口访问密钥）放进任何 `NEXT_PUBLIC_*` 变量。`NEXT_PUBLIC_*` 会进入浏览器，不能放密钥。缺少必要配置时，流量看板应该显示“尚未配置”的安全状态，而不是报错或泄露信息。

按当前仓库规则，client launch strict gate 默认要求 owner traffic dashboard 的 hostname、zone 和访问保护已配置。如果派生项目决定公开上线时完全不提供 `/ops/traffic`，要先单独调整 strict gate 和证明口径，不能只在文档里说“可不配”。

## 验证命令

最小 Cloudflare 构建证明：

```bash
pnpm build
pnpm website:build:cf
```

完整 release proof：

```bash
pnpm release:verify
```

`pnpm release:verify` 本身会串起构建、Cloudflare 构建、Wrangler dry-run 和相关 proof。已经跑完整 release proof 时，通常不需要再把前两条当成同一轮必跑命令重复执行；除非你就是要保留分步证据。
