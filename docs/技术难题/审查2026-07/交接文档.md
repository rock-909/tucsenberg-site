GitHub 一定会自动 retarget。

推荐顺序：

1. 分别审查并合并 #40、#41；两者都应确认 target 为 `main`。
2. #41 合并后，确认 #42 已明确 retarget 到 `main`。
3. 依次处理 #42 到 #50；每次合并后再次确认下一个 PR 的 base 和 diff。
4. 如果使用 squash merge 或 rebase merge，后代分支的父 SHA 会变化，必须
   restack 后代并重新核对 diff/CI，不能直接继续点 merge。
