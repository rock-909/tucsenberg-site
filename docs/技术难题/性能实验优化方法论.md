# 性能实验优化方法论

这是方法笔记，不是当前 Tucsenberg launch proof。当前上线证明仍以
`docs/项目基础/上线验证.md`、`docs/项目基础/发布验证.md` 和 `docs/项目基础/验证等级.md`
为准。

本文档记录本仓库当前采用的性能优化方法。派生项目可以直接复用这套流程，也可以按自己的页面和业务风险做裁剪。

这套方法的核心很简单：先用证据找最值得修的问题，只改一个变量，再用同一口径证明 before / after。收益不清楚就回滚，不把复杂代码留给后面的人收拾。

## 适用场景

适合用这套流程处理：

- Lighthouse 分数、`total-byte-weight`、LCP、TBT、TTFB 等指标治理；
- shared payload、全站 client cost、全局 CSS、字体、图片、prefetch、lazy loading；
- 关键页面的加载边界，例如首页、产品页、联系页；
- 会影响多个 route 的性能改动；
- 任何需要给客户、团队或后续 agent 留下可复核证据的性能实验。

不适合用这套流程处理：

- 纯文案修改；
- 没有用户可见收益的小整理；
- 只靠猜测的微优化；
- 需要牺牲业务内容、可访问性、SEO、i18n 或视觉体验的改动。

## 基本原则

1. 先排序，再实施  
   不要看到一个可疑点就改。先把候选列出来，比较影响范围、预期收益、风险和验证方式。

2. 一次只改一个变量  
   每轮只选择一个候选。不要顺手修第二个问题。否则 after 数据无法说明到底是哪一处带来收益。

3. fresh evidence 优先  
   Lighthouse 必须先 fresh build。旧报告只能作为线索，不能当当前代码的 after 证明。

4. 使用真实网络口径  
   `total-byte-weight` 按 Lighthouse transfer size / network payload 理解。不要把 treemap source size 当成真实网络节省。

5. 业务体验优先  
   不为了指标删除业务内容、砍掉 motion、降低 SEO、破坏可访问性、绕开 i18n，或放宽 Lighthouse 阈值。

6. 证据不成立就回滚  
   如果收益很小、数据不稳定、代码变复杂，或者带来行为风险，默认不保留。

## 标准工作流

### 1. 读取项目规则

在 starter 或派生项目中，先读当前项目自己的规则文件。不要依赖聊天记忆。

本仓库至少读取：

- `AGENTS.md`
- `docs/README.md`
- `docs/项目基础/替换顺序.md`
- `docs/项目基础/AI协作边界.md`

如果改动涉及 Next.js runtime、App Router、Server / Client Component、metadata、lazy loading、缓存、图片或字体，还要读取当前安装包里的 Next.js 文档。

如果改动触碰 UI、测试、安全、i18n、结构化数据等范围，按项目规则读取对应 rule 文件。

### 2. 核对当前状态

先确认：

- 当前 repo、branch、worktree 是否干净；
- 当前已有 proof 是否仍然适用；
- `.lighthouseci/` 是否有报告，报告是否来自当前代码；
- Lighthouse 配置和阈值是什么；
- 当前页面是否有已知 warning 或黄债。

常见检查面：

- `lighthouserc.js`
- `docs/技术难题/**`
- `.lighthouseci/lhr-*.json`
- 当前 git diff

如果现有 Lighthouse 报告来自已回滚实验、旧分支或旧构建，只能作为结构线索，不能当当前 before / after。

### 3. 做性能机会排序

先列候选表，再只选一个。

推荐表格：

| 候选 | 影响范围 | 预期收益 | 风险 | 验证方式 | 建议 |
| --- | --- | ---: | --- | --- | --- |
| 示例：Contact 表单加载边界 | `/contact` | 降低 JS transfer | 中：转化路径 | fresh build + Lighthouse + 表单 smoke | 可进入实验 |

排序时优先看：

1. shared payload / shared client cost：影响多个 route 的优先；
2. route-level transfer size 偏高页面：尤其接近或超过 Lighthouse warning 的 route；
3. LCP / render path：优先首页、产品页、联系页；
4. 图片、字体、prefetch、dynamic import、client/server boundary；
5. server waterfall：只有能证明改善 TTFB 或页面生成等待时才做。

每个候选都要说明：

- 为什么可能有可见收益；
- 影响哪些 route；
- 预计改善哪个指标；
- 是否会碰 motion、SEO、i18n、可访问性、内容或 UI foundation；
- before / after 怎么证明。

### 4. 写短 spec 和计划

进入代码前先写清楚本轮实验。

spec 至少包含：

- 目标；
- 不改什么；
- 用户可见行为；
- 验收标准；
- before / after 证据；
- keep / revert 标准；
- 最小验证命令。

计划至少包含：

- 修改范围；
- 测试或验证范围；
- 风险点；
- 停止条件。

### 5. 先跑 before

如果本轮要用 Lighthouse 证明收益，必须先 fresh build。

本仓库常用命令：

```bash
NODE_OPTIONS=--dns-result-order=ipv4first pnpm build
CI_DAILY=true NODE_OPTIONS=--dns-result-order=ipv4first pnpm website:lighthouse
```

如果派生项目只需要看少数页面，也可以先跑更小范围的 route 测量，但 closeout 仍应覆盖所有可能受影响 route。

记录 before 时至少保留：

- route；
- total-byte-weight；
- JS transfer；
- CSS transfer；
- document transfer；
- LCP；
- TBT；
- fetch time；
- 对应 LHR 文件或报告链接。

### 6. Test-first 实施

性能优化也要有行为保护。

改代码前先写或更新测试，证明：

- 本轮想保护的行为现在会被测试覆盖；
- 如果实现方式不满足目标，测试会失败；
- 失败原因不是拼写、mock 或测试环境错误。

测试不一定直接断言 Lighthouse 数字。它应该锁住用户行为、边界或加载策略。Lighthouse 负责证明性能收益。

### 7. 只改本轮变量

实施时不要扩大范围。

可以做：

- 把一个 client island 缩小；
- 延后加载一个非首屏、非关键路径组件；
- 移除一个已证明无用的 prefetch；
- 调整一个图片、字体或 CSS 边界；
- 修一个独立 server waterfall。

不要做：

- 顺手清理无关代码；
- 同时改多个 route；
- 同时改 CSS、JS、图片和文案；
- 删除内容换性能；
- 删除 motion 换小收益；
- 放宽 Lighthouse 阈值；
- 新增外部自动优化工具或循环。

### 8. fresh after 验证

改完后跑：

- focused tests；
- lint / type-check；
- React Doctor，如果改了 React、Client Component、hook、表单或渲染边界；
- Playwright smoke，如果影响关键用户流程；
- fresh build；
- Lighthouse after。

before 和 after 必须同一口径对比。不要拿旧构建、旧分支、不同 route 数量或不同统计方法混在一起比较。

### 9. 做 keep / revert 判断

保留标准：

- 指标改善明确，最好影响多个 route；
- 用户可见行为不退化；
- SEO、i18n、可访问性、motion、内容不受损；
- 复杂度增加合理；
- 验证链能证明收益。

回滚或 no-op 标准：

- 收益太小；
- 指标变化像噪声；
- 代码变绕；
- 行为、SEO、i18n、可访问性或视觉体验有风险；
- 需要牺牲 motion 或业务内容；
- 证据链不能证明收益来自本轮变量。

## Lighthouse 数据口径

优先使用：

- `total-byte-weight.numericValue`
- `resource-summary` 中的 `transferSize`
- LCP、TBT、FCP、Speed Index
- network requests 中的具体资源 transfer size

不要把这些当作 keep 证据：

- bundle analyzer 或 treemap 的 source size；
- 单次 Lighthouse 分数波动；
- 开发环境加载速度；
- 看起来更快这类主观判断。

如果 Lighthouse 分数变好，但 total、JS、CSS、LCP、TBT 都不能解释这个变化，默认当作噪声。

## Motion 边界

motion 可以有成本，但本项目不把删除 motion 当作默认性能路线。

可以做：

- 阻止 motion 无证据扩散；
- 不新增大体积动效依赖；
- 不为了装饰效果扩大 client cost；
- 保留 reduced-motion 约束；
- 记录 motion 成本和后续治理建议。

不要做：

- 删除页面转场；
- 移除全站 motion provider；
- 为了小收益砍交互动效；
- 把 motion removal 混进其他性能实验。

## 派生项目如何沿用

派生项目可以按下面顺序执行：

1. 复制本方法论到项目文档，或保留本 starter 文档作为引用；
2. 更新 route 列表和 Lighthouse 配置；
3. 用当前派生项目内容重新跑 fresh baseline；
4. 按候选表排序；
5. 每轮只做一个性能实验；
6. 写明 keep / revert 结论；
7. 把最终 before / after 写进项目自己的 proof 文档。

派生项目不要直接套用 starter 的旧数字。页面内容、图片、字体、第三方脚本、部署环境变了，性能事实也会变。

## 推荐 closeout 模板

每轮结束时记录：

```md

- 候选：
- 修改范围：
- 受影响 route：

- Lighthouse after：
- 指标变化：
- 行为验证：
- 风险：
- 判断：keep / revert
- 原因：
```

## 当前经验

本仓库已经验证过几个重要判断：

- shared payload 优化要优先，因为它能影响多个 route；
- Contact 页面是转化路径，优化时必须保留表单可填写、错误 fallback、Turnstile 和 smoke proof；
- 看起来合理的静态化不一定有网络收益，DataCard 静态表面实验就是无收益后回滚的例子；
- CSS / UI foundation 类优化潜在收益大，但会影响组件治理和 agent 分工，需要单独确认；
- 越到后期，剩余候选越容易变成小收益高验证成本，要更严格地筛选。
