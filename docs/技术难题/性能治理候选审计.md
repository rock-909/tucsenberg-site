## Outcome

The new performance governance rules are useful as a guardrail, but they do not
justify broad dynamic imports, broad memoization, or mechanical dependency
deletion.

Current repo evidence points to two real follow-up candidates:

1. shared motion and layout client islands that appear on every localized page;
2. one blog article server waterfall reported by React Doctor.

Everything else is either already narrow, intentionally deferred, or needs
manual starter-surface triage before removal.

## Scope and non-goals

This audit checked:

- Client Component boundaries;
- `React.lazy` and dynamic import usage;
- `motion/react` usage;
- `memo`, `useMemo`, and `useCallback` usage;
- React Doctor warning signals;
- Knip unused dependency, export, file, and type signals.

This audit did not:

- change production source files;
- delete dependencies;
- change Lighthouse thresholds;
- remove business content;
- add broad lazy loading;
- add broad memoization.

## Explicit non-findings

- There is no evidence for a repo-wide `next/dynamic` problem. Production code
  currently avoids `next/dynamic`; tests guard that behavior for several
  islands.
- There is no evidence for a repo-wide broad memoization problem. Existing
  memoization is concentrated in interactive stateful surfaces.
- There is no basis for a broad dependency purge. Knip output is noisy for this
  starter because profile fixtures, governance scripts, UI exports, generated
  helpers, and local skill scripts are not ordinary application entrypoints.
- The installed `next` package in this checkout does not contain
  `node_modules/next/dist/docs/`. Future Next.js runtime edits still need a
  current official or version-pinned API source before code changes.

## Recommendation

Do not broaden P1/P3/P3/long-term governance into immediate repo-wide fixes.

Use this branch to land the rule guardrails and the candidate audit. Then, if
you want to continue performance work, open separate lanes in this order:

1. P2 motion/shared-client-floor spike with before/after route evidence;
2. P2 blog article `Promise.all` micro-fix if behavior proof is straightforward;
3. P3 dependency-hygiene triage with starter-surface classifications.

That keeps the project aligned with the existing rule: no performance patch
survives without proof.
