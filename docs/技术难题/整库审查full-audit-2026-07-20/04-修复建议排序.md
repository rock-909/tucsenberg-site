# 修复建议排序

## 1. 边界

本文件只给修复顺序、最小改动方向和验收条件。本轮没有修改业务代码。是否进入修复轮、如何分支和合并，由 owner 另行裁决。

排序原则：先处理会造成页面不可用、丢单、安全门假绿或公开事实错误的问题，再处理证明覆盖和维护成本。每个 repair wave 都应独立绑定目标 SHA，不能用本审查的旧绿灯替代修复后的 fresh evidence。

## 2. Wave 1：Runtime、丢单和生产安全门

### 2.1 `FPH-001` Cloudflare/OpenNext 并发故障

- 最小修复方向：先做一个单变量 repair branch。优先测试关闭发布环境的 `cacheComponents`；如果已确认的 Next/OpenNext/Workerd 版本组合能直接修复，也可选择升级，但不要同时改多个变量。
- 必须补的门：同一产物的 8 路顺序请求、多轮并发请求、负载后的 `/api/health`，本地 Workerd 和绑定 SHA 的部署 preview 都要通过。
- 不能接受：只给现有 sequential smoke 加重试；它会隐藏 wedged isolate，而不是修复它。
- 完成条件：0 unexpected status、0 timeout、0 hung-request cancellation；正常 build、OpenNext、dry-run、page smoke、real-service canary 全部 fresh 通过。

### 2.2 `FPH-004` Provider receipt 假成功

- 根因位置：Resend 和 Airtable adapter 的外部回执边界。
- 最小修复方向：两处统一要求 `typeof id === "string" && id.trim().length > 0`；无效回执作为 provider failure。Pipeline 不再复制 provider-specific 判断。
- 必须补的测试：missing、empty、whitespace ID；邮件失败加无效 Airtable receipt；任一有效通道成功时保留现有 partial-success 合同。
- 完成条件：不存在 buyer success 但两个通道都没有有效 receipt 的路径；真实 canary 中 provider ID 能对应实际邮件或记录。

### 2.3 `FPH-005` Strict production gate 漏检危险开关

- 根因位置：`validateProductionRuntimeContract()` 和 production workflow 的有效配置传递。
- 最小修复方向：在现有合同中集中拒绝 `NEXT_PUBLIC_TEST_MODE=true`、`SECURITY_HEADERS_ENABLED=false`、`NEXT_PUBLIC_SECURITY_MODE=relaxed`；production workflow 必须传入或直接解析这些有效值。
- 运行时自卫：客户端 test mode 不能仅依赖发布脚本，production runtime 也应拒绝或忽略。
- 完成条件：三项危险值单独和组合都 hard fail，安全 fixture 继续通过；部署响应头和真实 Turnstile 询盘通过。

## 3. Wave 2：公开事实和关键可达性

### 3.1 `FPH-002` TB-FB 保修范围

- 最小修复方向：RFQ 通用 reassurance 改成不声明统一期限的真实表述；About/OEM 明确限定三年材料和工艺保修只适用于标准耐用品。
- 不建议：新增产品保修 factory 或大规模 schema。当前问题只需要收口现有公开文字。
- 完成条件：TB-FB 进入 RFQ 时不会看到三年材料和工艺保修；Warranty 页继续保留 shelf life 与 warranty 区分。

### 3.2 `FPH-003` Aluminum 卡片配置错误

- 最小修复方向：只改首页 Aluminum 卡片这一行，使用已有 Aluminum truth 中的叠板、立柱、角柱或按开口定尺能力。
- 完成条件：全库 curve/gable-end 的正面产品能力只指向 ABS；首页链接和 product slug 对账继续通过。

### 3.3 `FPH-006` 移动端表格键盘滚动

- 根因位置：共享横向 scroll owner。
- 最小修复方向：让真实 overflow container 进入顺序焦点，增加可见 focus ring 和必要 table context。产品页和 Markdown table wrapper 一起处理；同类首页 wrapper 同轮覆盖。
- 完成条件：Pixel 5 宽度下 Tab 能到达四个表格，方向键改变 `scrollLeft`，焦点清晰，Axe 不再报告 `scrollable-region-focusable`。

## 4. Wave 3：发布和门禁可信度

### 4.1 `FPH-007` CI Cloudflare platform signal

- 最小修复方向：Cloudflare CI 的 Next/OpenNext build 使用 canonical platform vars。
- Sourcemap 分析：如果 client-boundary 检查依赖 sourcemap，把它改成诚实命名的独立分析 build，或让它能检查 production artifact。
- 完成条件：`.next` 和 `.open-next` 均显示 sourcemaps off、`images.unoptimized=true`，CI 仍能完成 client-boundary proof 和 Wrangler dry-run。

### 4.2 `FPH-010` Comments-only source gate

- 最小修复方向：删除已经被真实 build/dry-run 覆盖的 raw string checks。剩余唯一规则用 JSONC、YAML、package JSON 或最小 AST 读取实际字段和命令。
- 完成条件：comments-only fixture 必须失败，真实 source 正例和行为证明继续通过。

### 4.3 `FPH-011` Production cancellation

- 最小修复方向：production run 禁用 `cancel-in-progress`，让 deploy 和 post-deploy verification 串行完成；preview 可按需保留取消。
- 完成条件：解析型 workflow test 锁定 production 不取消；实际授权发布中第二次 dispatch 排队。

### 4.4 `FPH-012` Daily flaky-green

- 最小修复方向：Daily 设置已有的 `CI_FLAKE_SAMPLING=1`，使 retries 为 0；普通 PR smoke 是否保留 retry 可单独决定。
- 完成条件：Daily 环境配置测试得到 `retries=0`；下一次 intermittent failure 会把 workflow 变红。

### 4.5 `FPH-013` Mandatory concurrency smoke

- 最小修复方向：把并发请求和 `x-middleware-set-cookie` 断言放进一个 mandatory path。优先增强 `deployed-smoke`，同时保留每个请求的有限 retry。
- 完成条件：挂起首个响应时，强制 smoke 已启动其他路由；cookie header 泄漏会失败；manifest/workflow contract 能证明该步骤强制执行。

## 5. Wave 4：恢复、测量和事实 owner

### 5.1 `FPH-008` Partial-success 关联键

- 最小修复方向：把现有 `referenceId` 放入 owner email data、正文或主题、Resend tag 和 success log。
- `requestId` 处置：有真实生产来源就接入，没有消费者就删除测试专用 seam。
- 完成条件：Buyer response、owner email、provider success/failure log 共享同一 `referenceId`，Airtable 失败时 owner 能确定性回填。

### 5.2 `FPH-009` Lighthouse 路由清单

- 最小修复方向：换成 16 个 canonical 无前缀路由，保留现有三次聚合和阈值。
- 完成条件：`CI_DAILY=true pnpm website:lighthouse` 直接请求 16 页，0 locale redirect；URL 集与 current route truth 有小型合同测试。

### 5.3 `FPH-014` 产品事实多 owner

- 删除优先：不必要的精确产品数量和重复数字先改成不依赖数量的表述。
- 派生优先：少数必须显示的数字从已有 typed product value 传给 diagram/metadata presenter。
- 不建议：新建产品 CMS、通用产品事实平台或大范围 schema migration。
- 完成条件：改变一个 TB-BW 高度值会让所有保留表面一起变化或一起失败；增加第六个产品不会留下 active `five` 文案。

## 6. Wave 5：删除死表面

### 6.1 `FPH-015` 五个 stale public assets

- 删除前检查：确认没有外部直接链接依赖这五个路径。
- 修复方向：移除五个文件，更新 `public/images/README.md`，保留当前 live logo/OG 和 owner intent 未明确的 square logo。
- 完成条件：Production reference scan 清零；旧直链不再返回陈旧内容；当前 logo 和 OG 正常。

### 6.2 `FPH-016` 两个 retired compatibility symbols

- 修复方向：删除 `getContactCopy()` 和只为它存在的测试；测试改到 live `getContactCopyFromMessages()`。让 `MAX_LEAD_PRODUCT_NAME_LENGTH` 直接持有其限制，删除 company alias 和 barrel export。
- 完成条件：`rg` 清零两个退役符号；Contact tests、lead schema/handoff、Knip、type-check 全过。

## 7. Owner-deferred 和凭据项

以下项目不能混入代码 repair wave 的完成声明：正式域名、全部 PDF 修复、公开电话和产品照片、管坝 MOQ、法律/联系页签字，以及 Cloudflare/Turnstile/Resend/Airtable/GSC/CrUX 的账号侧证明。

Repair wave 可以先完成不依赖它们的代码和门禁修复，但公开上线结论必须等 owner 输入和真实部署证明补齐后重新审查。
