# 审查过程复盘

## 1. 本轮方法

本轮按 `repo-health-audit` 的证据管线执行：

1. 先用 `git ls-remote` 锁定 `origin/main@9ab5f6c4f158281fd16c987e0cdd02622919d90e`。
2. 对齐本地 HEAD、`origin/main`、merge-base 和 worktree。
3. 先跑 runtime/build 基线，再走三条关键链路。
4. 八条车道盲派，车道只写自己的报告，不下整库终局。
5. 主线程对车道完成声明、证据路径、命令输出和 Git 状态做机械对账。
6. 只允许两条独立车道各自报告的同根因直接合并；其余单源候选都做以推翻为目标的反证。
7. 最后归一化 findings、rejected、Blocked、Not-run 和 failed，生成 65 项 Coverage Map。

审查期间没有修改业务代码。构建和测试产生的写入只落在 gitignored 产物或工作证据目录。

## 2. 哪些检查最有信号

### 2.1 Node 与 Workerd 的同条件并发对照

单独看 build、dry-run、CI 和 sequential smoke 都是绿色。只有把同一 SHA、同一路由、同一超时和清空代理后的 Node/Workerd 结果放在一起，才定位到 Cloudflare runtime 路径的并发故障。这是本轮最高价值的运行证据。

### 2.2 Provider malformed receipt 探针

现有测试数量很多，但没有覆盖 missing Airtable ID 和 whitespace Resend ID 的组合。两个独立车道分别从不同入口复现后，主线程再次跑到 `success=true` 的决定性分支，避免把类型声明或 Promise fulfilled 当作真实交付。

### 2.3 Gate mutation

对 production contract、comments-only source gate、workflow cancellation 和 smoke mapping 做小型 mutation，比继续增加 happy-path 测试更快暴露假绿：

- 三项危险 production switch 被 strict gate 接受。
- Required Cloudflare 字符串只留在注释中，source gate 仍接受。
- Production deploy 和 post-deploy verification 分成两个 job，却允许新 run 取消旧 run。
- 手工 smoke 有并发和 cookie header 断言，强制 smoke 没有。

### 2.4 全部二进制资产直接检查

代码搜索看不到 PDF 页面画布、裁切、字体、可提取文字和视觉布局。本轮对 6 个 PDF、8 个图片/SVG 全部做了直接检查，确认了 R'12 延期台账和 5 个 dead public assets。PDF 问题没有因为 owner 延期而跳过检查，也没有被错误包装成本轮 finding。

## 3. 哪些信号容易误导

### 3.1 早期依赖环境

本机安装的依赖最初落后于 lockfile。第一次 type/lint/test/build 即使为绿，也不能代表目标提交。依赖同步后重新建立基线，旧日志全部标为 stale。

后续审查应把 lockfile 与 `node_modules` 版本核对放到 required commands 前面，避免先产生一批无效证据。

### 3.2 CI 全绿

Exact-SHA CI 的 6 个 job 都成功，但它没有证明 canonical Cloudflare platform build，也没有覆盖 Workerd 并发。Daily workflow 也能在 `1 flaky` 后保持 success。

以后应把 `测试存在`、`CI 执行`、`release mandatory`、`production deployed proof` 分开记录，不再使用一个 `全绿` 标签覆盖四种合同。

### 3.3 未绑定 SHA 的公开 preview

公开 preview 在并发下也出现 500/timeout，但页面没有可靠 build identity。它只加强故障模式的可信度，不能证明目标提交当前部署状态。

### 3.4 静态模式和文件大小

`/api/health` 被 prerender、文件超过行数阈值、依赖有可用更新、Knip autoconfig 报 unused，都只是线索。没有对应行为、边界或消费者证据时，不应直接升级为 finding。

## 4. 子任务对账

八条 lane 报告和三条 chain 报告最终都存在，决定性引用路径全部通过存在性检查。主线程没有直接接受子任务的 `完成` 或 `全绿` 声明。

Lane 07 暴露了一个具体流程问题：worker 最初声称 canonical 报告和兼容镜像完全一致，第一次机械检查却得到 `cmp_exit=1`。修正后 fresh check 为 `cmp_exit=0`。本轮最终报告保留了这段差异；镜像没有被算作第二条独立车道来源。

这个问题说明完成声明至少要核对三件事：文件存在、内容一致、Git 状态符合写入边界。后续 lane contract 应把 `cmp`、`test -e` 和 `git diff --name-only` 的实际输出作为 mandatory closeout，而不是只要自然语言汇报。

## 5. 反证收口

最终 16 个 findings 中：

- 2 个由两条独立车道分别报告同一根因。
- 14 个为单源 finding，全部完成推翻式反证。

本轮最重要的裁决有三项：

1. 保留 `FPH-L00-002`，但把表述收窄为 `CI 未使用 canonical Cloudflare platform signal`。Lane 06 对 preview/production 角色差异的反驳是对的，却没有推翻真实 artifact 设置差异。
2. 拒绝 `FPH-L03-003`。静态 `/api/health` 不能证明 provider readiness，但当前 authoritative proof model 本来也没有要求它承担该职责。
3. Daily 的 FAQ/UI 原因没有复现，只保留 green-with-flaky 的门禁 finding。

归一化结果为：16 findings、36 rejected candidates、12 Blocked、5 Not-run、1 failed。

## 6. 证据保留策略

工作证据继续放在：

```text
.context/audits/full-audit-2026-07-20/
```

最终报告不把该 gitignored 目录当正式证据根。只有决定性 runtime、反证、关键链路、覆盖清单和二进制检查被复制到本报告目录的 `evidence/`，并生成 `manifest.sha256`。

没有复制完整 GitHub runner 日志、env/proxy dump、探针源码或凭据文件。保留的文件只含公开 URL、测试 fixture、公开 site key、变量名或合成数据，没有真实 secret 值。

## 7. 下轮改进

1. Preflight 加入 lockfile 与已安装版本一致性检查，未通过前不跑基线。
2. Cloudflare artifact 增加 build SHA 或可验证 identity，避免公开 preview 只能作辅助证据。
3. Mandatory release proof 加入并发 Workerd 探针，减少审查时额外搭建诊断工具。
4. Lane closeout 固定输出机器可读 counters、证据路径清单和 Git diff，而不是只写自然语言完成声明。
5. Coverage map 由 preflight 单元自动闭合，最终仍由主线程检查 65 行、固定 7 列和合法 mode。
6. Final report 在生成后统一跑 findings validator、audit config validator、引用存在性、严重度对账、文档清单和 anti-trope 检查。

本轮到此只完成审查、证据收口和修复排序。业务修复必须另开 repair wave。
