## PR-G：失效防线与守护缺口（最高优先级——这些是"看起来有防护、实际没有"）

### G1. Semgrep XSS 主防线是死规则

- **证据**：`semgrep.yml:3` 用 `pattern-regex: 'dangerouslySetInnerHTML=\{\{__html:\s*\$[A-Za-z_][A-Za-z0-9_]*\}\}'`——作者把 Semgrep 元变量语法（`$VAR`）误写进正则，`\$` 在正则里匹配**字面美元符号**。真实代码 `dangerouslySetInnerHTML={{__html: userInput}}` 不含 `$`，**永远不命中**。同时漏 `{{ __html: ... }}`（带空格）、`obj.prop`、模板串形态。
- **修法**：删这条手写正则，改为真 Semgrep pattern（参照官方 registry `p/react` 的 `react-dangerouslysetinnerhtml` 写法：pattern 匹配 `<$X dangerouslySetInnerHTML=... />` 与对象形态），或直接在 CI 引入 `p/react` 官方规则集并把本地这条删掉。改完必须写一个**会命中的反例**验证规则活着（临时文件测一下再删）。
- **同 PR 顺手修**（同类根因：手写 pattern 弱）：`semgrep.yml:24-28` `unsafe-eval-usage` 只匹配单参 `Function($ARG)`，漏 `new Function("a","body")`——改 `new Function(...)`；`semgrep.yml:14` `hardcoded-api-keys` 值字符集 `[a-zA-Z0-9]{20,}` 漏含 `-`/`_` 的 token（Cloudflare token 常含），扩字符集。
- **保留**：`semgrep.yml:469-573` 的 5 条 repo 契约规则（safeParseJson 强制、raw proxy header 禁读等）是官方 registry 没有等价物的好围栏，**不动**。
- **注意**：规则等级/清单变化会撞 `tests/architecture/ci-workflow-contract.test.ts` 的 `ERROR_SEMGREP_RULE_IDS` 精确锁（:16-25, :89），同步更新。此项与姊妹篇 PR-A2（开放重定向规则升 ERROR）、PR-D4（15 条 WARNING triage）同文件，**建议合并执行**。
- **验证**：`semgrep scan --error --severity ERROR --config semgrep.yml src` + 反例命中测试 + ci-workflow-contract 测试绿。

### G2. Turnstile 把 Cloudflare 服务端故障算在买家头上（丢询盘）

- **证据**：`src/lib/security/turnstile-errors.ts:1-5` 的 `TURNSTILE_SERVICE_FAILURE_CODES` 只含自造码（`not-configured`/`network-error`/`timeout`）。Cloudflare siteverify 返回 `error-codes: ["internal-error"]` 时落到 `src/lib/security/lead-turnstile.ts:116-120` 的 failed 分支 → 400 `TURNSTILE_REJECTED`。官方错误码表明文：`internal-error` = 服务端内部错误，**"The request can be retried"**（developers.cloudflare.com/turnstile，siteverify 错误码节）。
