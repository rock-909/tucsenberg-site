## PR-A：两个真 bug（最高优先级）

### A1. 翻译缓存 revalidate 单位错误（毫秒当秒用，30 分钟变 20.8 天）

- **证据**：`src/lib/i18n/load-messages.ts:36`
  ```ts
  const revalidate = () => (isDev() ? 1 : MONITORING_INTERVALS.CACHE_CLEANUP);
  ```
  `MONITORING_INTERVALS.CACHE_CLEANUP` 定义于 `src/constants/performance-constants.ts:57` = `BASE_NUMBERS.THIRTY * TIME_UNITS.MINUTE` = **1,800,000 毫秒**（语义："性能监控缓存清理间隔: 30分钟"）。它在 `load-messages.ts:102` 被传给 `unstable_cache` 的 `revalidate`，该参数单位是**秒**（先读 `node_modules/next/dist/docs/` 确认当前版本签名）。实际失效期 ≈ 20.8 天。
- **根因类**：语义盲借用（跨文件、跨语义、跨单位），与 2026-07 全库审计的 G2 类同构，属漏网样本。
- **修法**（2026-07-11 修订，见姊妹篇《官方对标修复执行文档.md》决策项 4）：后续官方对标审查发现该 `unstable_cache` 包装本身近零收益——仅本地 dev 生效（CI/构建/Cloudflare 全绕过，`load-messages.ts:111-116`）、缓存对象是打包内静态 JSON import（模块系统已缓存）、且官方已判 unstable_cache 被 `use cache` 取代。**首选修法：直接删除 unstable_cache 包装**（连带清理 `cache-tags.ts` 中绑定的 i18n tag），而不是修单位。若 owner 未批准删除，退回次选：就地定义带单位后缀的常量（如 `I18N_MESSAGES_REVALIDATE_SECONDS = 30 * 60`），注释写明"unstable_cache revalidate 单位为秒"。无论走哪条，**都要全仓 grep `unstable_cache` 和 `revalidate` 核查同类单位错配**，再 grep `MONITORING_INTERVALS.` 的其他消费者确认没有第二处借用。
- **验证**：`pnpm type-check && pnpm test`，并跑 `pnpm build` 确认运行时改动无回归。

### A2. 开放重定向检测两头落空

- **证据**：`eslint.config.mjs:341` 关闭 `security-node/detect-dangerous-redirects`，注释称"已迁移：semgrep nextjs-unsafe-redirect 规则覆盖"。但 `semgrep.yml:39` 的 `nextjs-unsafe-redirect` 是 `severity: WARNING`，而 CI（`.github/workflows/ci.yml:167`）只跑 `semgrep scan --error --severity ERROR`。→ 两边都没在执行。
