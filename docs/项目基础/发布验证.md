# Release Proof

Use this for Tier A or production-sensitive changes.

## Command

```bash
pnpm release:verify
```

It runs the canonical release sequence through `node scripts/starter-checks.js release-verify`.
The authored source of truth for that sequence is `scripts/quality/release-proof-manifest.js`.
The visible command block below is checked against the manifest so the runbook,
runner, and tests do not drift.

Local release proof is not public launch proof. Public launch still needs `PUBLIC_LAUNCH_STRICT=true node scripts/starter-checks.js validate-production-config`, deployed lead canary, and manual launch gate.

Pull request CI does not prove the Cloudflare Free gzip budget, because the
Wrangler dry-run needs Cloudflare credentials and is skipped for
`pull_request`. Before merging a Cloudflare-size-sensitive PR, record either a
`workflow_dispatch` CI run or a local `pnpm release:verify` run against the
exact head SHA.

Lighthouse remains manual performance proof. It is not part of default CI, git
hooks, or `pnpm release:verify`; run `pnpm build && pnpm website:lighthouse`
only when a change makes a page-performance claim.

## Rules

- Run build before `website:build:cf`; never run them in parallel.
- `cf-static-asset-headers` proves OpenNext copied `public/_headers` into
  `.open-next/assets/_headers` with the long-cache rule for `/_next/static/*`.
- The Wrangler preview dry-run is the source-checkout Cloudflare Workers Free
  runtime budget proof: keep Worker upload below **3000 KiB gzip**, with a
  preferred target below **2700 KiB**.
- This release proof covers the current single catalog site only.
- Release-proof failure means not release-proven.
- Release-proof is technical evidence, not automatic public launch approval.
- Public launch still needs deployed smoke, real service canary, and owner signoff.

## Dirty worktree

dirty worktree rule:

If unrelated dirty files exist, report:

1. targeted proof for changed seam;
2. clean branch proof for whole-repo claims.

Do not collapse them into one “green” claim.

When the change touches single-site truth, name the exact seam: `src/config/single-site.ts`, `src/config/single-site-page-expression.ts`, or `src/config/single-site-seo.ts`.

## Extra launch canary

For public launch:

```bash
POST_DEPLOY_TEST=1 PLAYWRIGHT_BASE_URL="$DEPLOYED_BASE_URL" pnpm exec playwright test tests/e2e/smoke/
```

This proves the deployed path only when the environment uses real deployed credentials and the resulting Airtable/Resend/Turnstile state is checked.

This extra step is a manual launch gate; a GET deployed smoke does not prove Airtable or real lead delivery.
The current Playwright canary verifies the Airtable record (`recordCreated`); `ownerNotified` / owner notification still needs manual target-system confirmation.

## Manual Airtable phone column proof

Use this when a change must prove the real Airtable `WhatsApp / Phone` column receives a normalized phone value through the production write path. The workflow is manual-only; it does not run on push or pull request CI.

```bash
gh workflow run airtable-phone-proof.yml --ref <exact-branch> -f airtable_table_name=Contacts
gh run watch --exit-status
```

After the run finishes, confirm the workflow run `headSha` equals the exact commit SHA you intended to prove. A green default CI run on that SHA is not the same as a green phone proof run.

The workflow checks out the dispatch commit (`github.sha`), so the SHA recorded in the workflow summary is the SHA of the code that actually ran the proof.

The workflow summary records only ref, SHA, table name, and pass/fail. It does not print record ids, phone values, API responses, or secrets. The proof test cleans up its own Airtable record.

### When the proof fails with 404 NOT_FOUND

A `404 NOT_FOUND` from Airtable means the Base or table is not reachable with the credentials in the run, or the table name does not match. Do not guess table names or substitute mocks.

1. In GitHub Actions, confirm `AIRTABLE_BASE_ID` and the PAT behind `AIRTABLE_API_KEY` can access the intended Base.
2. Confirm the dispatch input `airtable_table_name` (or a future `AIRTABLE_TABLE_NAME` secret) matches the real table name in that Base.
3. Confirm the table has an exact column named `WhatsApp / Phone`.
4. Re-run the workflow on the exact commit SHA you intend to prove:

```bash
gh workflow run airtable-phone-proof.yml --ref <exact-branch-or-sha> -f airtable_table_name=<actual-table-name>
gh run watch --exit-status
```

A green default CI run on the same SHA is not a substitute for a green phone proof run.

### Recovery prerequisites (Cluster 3A C2, 2026-07-18)

GitHub Actions needs a PAT and Base ID that can reach the real Airtable Base, plus the actual table name. Prefer storing the table name as an `AIRTABLE_TABLE_NAME` repository secret or passing it explicitly on each dispatch. The target table must expose the exact column `WhatsApp / Phone`.

Current state: the phone proof workflow checks that `AIRTABLE_API_KEY` and `AIRTABLE_BASE_ID` exist; it does not yet require `AIRTABLE_TABLE_NAME`. Cloudflare preview can confirm a secret name exists but cannot read its value, so preview secret presence alone is not evidence that GitHub Actions and deploy use the same Base, table, or column.
