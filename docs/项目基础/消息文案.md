# Messages

Physical message packs are the authoring truth. There is no generated
compatibility layer under `messages/{locale}/`.

## Fixed graph

The Tucsenberg site always composes messages in this order:

```text
base -> b2b-lead -> catalog
```

The directories are:

```text
messages/base/{locale}/messages.json
messages/profiles/b2b-lead/{locale}/messages.json
messages/profiles/catalog/{locale}/messages.json
```

`b2b-lead` and `catalog` are fixed ownership layers. They are not selectable
runtime profiles.

Leaf-key ownership is mutually exclusive across the three packs. A later pack
must not redefine a leaf owned by an earlier pack, and must not place a leaf
under (or above) another pack's leaf path (for example `foo` vs `foo.bar`).
Runtime composition still walks `base -> b2b-lead -> catalog`, but
`pnpm content:check` fails on exact or parent/child path conflicts. That keeps
TypeScript intersection typing aligned with runtime composition.

Runtime, Storybook, and TypeScript key types all read these packs through the
shared composition helper `src/lib/i18n/composed-messages.ts` (and the async
loader for server runtime). Adding a locale needs locale config plus the three
physical packs and matching MDX/SEO/runtime proof — not a generated copy.

## Ownership

| Surface | Owner |
| --- | --- |
| cross-page infrastructure: theme, accessibility, cookie, structured data, API/errors, legal and email reference copy | `messages/base/**` |
| contact, RFQ, inquiry and shared lead-form copy | `messages/profiles/b2b-lead/**` |
| current site shell and catalog experience: navigation, footer, home, catalog, OEM landing and product-page UI | `messages/profiles/catalog/**` |
| page prose and page SEO | `content/pages/en/*.mdx` |
| product specifications and product copy | `src/constants/tucsenberg-product-page-*.ts` |

Long page prose does not belong in message JSON. Product specifications do not
belong in message packs.

## Transactional email copy

Website messages and transactional email copy are related but separate:

- next-intl reads the website message graph;
- the lead pipeline sends owner/buyer email through Resend using plain
  `html` / `text` built by `src/lib/email/runtime-email-content.ts`;
- `emailTemplates` in the base message pack is the authoring/reference
  namespace for transactional email copy;
- `src/emails/email-copy.ts` is the runtime email copy API used by that
  production content builder and by subject helpers.

There is no React Email template runtime in this site.
runtime email rendering still defaults to English, matching the current site.
